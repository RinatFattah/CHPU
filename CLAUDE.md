# CLAUDE.md

## Project Summary

`ai_3d_pipeline` is a small Python demo pipeline that turns a natural-language part description into:

1. Blender Python code generated by an LLM
2. an STL file exported by Blender via `blender-mcp`
3. a simple contour-style G-Code file derived from STL bounds

The repository is meant for demos and operator handoff, not as a production-grade CAM system.

## Key Entry Points

- `server_3d.py` - main FastAPI service and the best entry point for end-to-end testing
- `app_streamlit.py` - manual UI for demos
- `auto_3d.py` - interactive CLI for `text -> STL`
- `pipeline_3d.py` - interactive CLI for `text -> STL -> G-Code`
- `stl_to_gcode.py` - standalone STL to G-Code helper
- `config.py` - default config values and YAML loading

## Environment and External Dependencies

- Python 3.10+
- Blender running with `blender-mcp` reachable on `BLENDER_HOST:BLENDER_PORT` (default `localhost:9876`)
- OpenAI-compatible API key in config or environment
- Optional Streamlit for the UI
- Optional FreeCAD for the experimental path in `stl_to_gcode.py`

Do not assume the pipeline is runnable in CI or on a clean machine without Blender/OpenAI access.

## Config Rules

Config precedence is:

1. defaults in `config.py`
2. environment variables
3. YAML loaded with `config.load(...)`

Supported operator workflow is to copy `config.example.yaml` to `config.yaml` and run scripts with `--config config.yaml`.

Important details:

- `config.yaml` is intentionally ignored by git because it may contain secrets
- `OUTPUT_DIR` may contain `~`, which is expanded in `config.load()`
- `server_3d.py`, `auto_3d.py`, `pipeline_3d.py`, and `stl_to_gcode.py` now support `--config`
- `app_streamlit.py` accepts config via `streamlit run app_streamlit.py -- --config config.yaml`

## Architecture Notes

There is deliberate code duplication across CLI, API, and UI flows. The most reusable implementation currently lives in `server_3d.py`:

- `generate_valid_code()`
- `run_in_blender()`
- `read_stl_bounds()`
- `generate_gcode()`

`app_streamlit.py` imports these functions directly from `server_3d.py`.

If you refactor, preserve the current behavior first, then extract shared logic into a dedicated module rather than re-implementing it in multiple places.

## Operational Gotchas

- Blender communication is raw socket JSON, not HTTP.
- The LLM sometimes returns fenced code or invalid Python; `clean_code()` and syntax validation are there for a reason.
- Reasoning-heavy models may need a much higher `OPENAI_MAX_TOKENS`, otherwise code generation can return an empty or truncated response.
- The generated G-Code is simplistic and based on STL extents, so avoid describing it as full CAM.
- `stl_to_gcode.py` contains an unfinished FreeCAD path and a practical fallback generator. Treat the fallback as the default working behavior unless proven otherwise.
- Generated `.stl` and `.gcode` files are ignored by git and should stay out of commits unless the user explicitly asks otherwise.

## Safe Verification

Use lightweight checks that do not require external services unless the user explicitly wants a live run:

```bash
python -m compileall .
python server_3d.py --help
python auto_3d.py --help
python pipeline_3d.py --help
python stl_to_gcode.py --help
```

Live validation requires:

1. Blender MCP running
2. valid OpenAI credentials
3. a local `config.yaml`

Then the safest smoke test is:

```bash
python server_3d.py --config config.yaml
curl http://localhost:8765/health
```

## Editing Guidance

- Keep documentation in Russian unless the user asks for English.
- Prefer `README.md` as the primary handoff document.
- When changing runtime behavior, update both `README.md` and `config.example.yaml` if the operator contract changed.
- Avoid changing prompt wording casually; small edits can materially change generation quality.
- Avoid committing secret-bearing files such as `config.yaml`.
