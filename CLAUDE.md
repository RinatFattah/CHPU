# CLAUDE.md

## Project Summary

CAM pipeline: **CAD model → G-Code** for a 3-axis CNC mill, generated by FreeCAD's
Path/CAM workbench (headless). No toolpath is supplied — it is computed from the model.
Default output is **staged roughing** ("like a machinist"): RoughContour (perimeter
material between the FILLED silhouette and the stock box — exists even at
`STOCK_MARGIN` = 0, since stock is always at least a box and corner rounds/protrusions
leave material; wide margins ≥ 2 tool Ø → Adaptive ring grown one tool Ø past the
stock edge — else StockToLeave leaves a 0.5 mm skin standing at the stock boundary;
narrow/uneven → MULTI-LOOP Path Profile passes outside-in (loop count from the max
material distance to the silhouette via distToShape over material vertices + bbox
corners), which need no helix entry) → RoughHole1..N (through-cutouts
of ARBITRARY shape = inner wires of the part silhouette, nearest-first) →
RoughFace1..N (one op per upward-facing planar face, top-down) → RoughSlope1..N
(one op per slanted/curved face with upward normal component: terraced Path Surface,
Multi-pass + DepthOffset=allowance; the face is usually NARROWER than the tool, so
CutPattern=ZigZag + BoundaryAdjustment=tool radius + BoundaryEnforcement=False are
REQUIRED — the default Offset pattern insets by the tool radius and yields an empty
path). Stages 1–3 are Adaptive ops over explicit non-overlapping 2D regions, with
`ROUGH_ALLOWANCE` stock left on walls (StockToLeave / Profile OffsetExtra) and floor
(FinalDepth). The contour region subtracts the FILLED silhouette (outer wires only):
subtracting the holed silhouette drops through-cuts into the contour zone → double
machining. Down/side-facing or covered faces are skipped (warned, second setup);
material unreachable from above (overhangs) is skipped —
that's a second setup, not something to cut through. Silhouette sections MUST be
built with `Part.makeFace(wires, "Part::FaceMakerBullseye")` — making a Face per
wire fills cutouts and the union loses the holes. A Path Surface finishing
pass follows by default (`--no-finish` / `FINISH: false` disables). Stock = part
bbox + margins (`STOCK_MARGIN`, default 12 = 2 tool Ø) OR an arbitrary solid from a
file (`STOCK_FILE` / `--stock`): the stock file must be in the SAME coordinate
system as the model — the worker records the part's orient/origin transforms in a
journal and replays them on the stock. FreeCAD 1.1 has no `Stock.CreateFromExisting`,
but assigning a plain `Part::Feature` to `job.Stock` works (ops only read
`job.Stock.Shape`); the contour zone is then built from the stock's real silhouette.
An INVALID stock solid silently breaks Adaptive (empty paths) — the worker attempts
a sew+makeSolid repair and warns. Stock/part/tool dimensions are
embedded as `(Stock: ...)`/`(Part: ...)`/`(Tool: ...)` header comments in the
G-Code.
The customer's source parts are Siemens NX `.prt`; the bridge is STEP export from NX
(FreeCAD cannot read `.prt`, no open-source readers exist).

CRITICAL Adaptive gotcha: regions must be explicit planar faces. Passing model faces
as `Base` yields open projected contours on real parts (`Path.Area: ccurve not closed`)
and Adaptive silently returns an empty path. Adaptive also needs ~2 tool diameters of
material width for its helix entry — narrow perimeter slivers (e.g. corner rounds at
`STOCK_MARGIN` 0) get a partial/empty path, hence the Profile fallback for the contour.
Orientation gotcha: the "features up" flip must test hole OPENNESS (`isInside` probes
above/below the cylinder), not "cylinder near the bottom" — a through-window in a
bottom plate would otherwise flip the part upside down and make everything but the
window an unreachable overhang.

An earlier LLM/Blender text→STL demo pipeline was removed from the repo (July 2026);
it lives in git history if ever needed.

## Files

- `run_cam.py` - CLI entry point: `python run_cam.py model.step [out.gcode] [--config config.yaml]`
- `freecad_cam.py` - host side: locates `freecadcmd`, passes params via temp JSON
  (env var `FREECAD_WORKER_PARAMS`), parses `[worker]` output markers.
- `freecad_worker.py` - runs **inside** FreeCAD's interpreter: model → solid →
  origin normalization → Path Job → Surface op → postprocessor → G-Code.
- `config.py` - parameter defaults + YAML loading (`--config`).
- `README_CAM.md` - operator-facing parameter reference. **Keep in sync** when
  changing CAM params. `README.md` - main handoff doc.

## Key Facts

- Input formats: STEP/IGES/BREP read as exact BREP solids (units from file,
  `STL_SCALE_TO_MM` ignored); STL/OBJ via mesh→faceted-solid conversion; `.prt`
  fails with a clear message (both in `run_cam.py` and the worker).
- FreeCAD runs as a **separate headless process** (`QT_QPA_PLATFORM=offscreen`);
  its Qt/OpenCASCADE never load into the main Python. Params go through a JSON file
  referenced by env var — NOT argv (freecadcmd executes every file argument as a script).
- Worker is executed by freecadcmd as a plain script (not `__main__`) — hence the
  `if os.environ.get("FREECAD_WORKER_PARAMS"): main()` guard at the bottom.
- `ORIGIN` normalization (default `corner-top`: X0Y0 = bbox corner, Z0 = top) exists
  because NX parts are often exported in assembly/machine coordinates, meters away
  from zero — without the shift the first G0 sends the tool far away.
- FreeCAD on this machine: extracted AppImage at
  `~/freecad-appimage/squashfs-root/usr/bin/freecadcmd` (v1.1.1). apt has no FreeCAD
  on Ubuntu 24.04; the snap is broken headless (Qt symbol mismatch vs kf6-core24).
- FreeCAD's CAM has **no lathe/turning operations** — turning was prototyped and
  removed; don't promise it.

## Editing Guidance

- Keep documentation in Russian unless asked otherwise; `README.md` is the primary
  handoff doc, `README_CAM.md` the parameter reference.
- When changing runtime behavior or params, update `README.md`, `README_CAM.md`
  and `config.example.yaml` together.
- Generated `.stl`/`.gcode` are gitignored; keep them out of commits.
- `config.yaml` may contain operator-specific settings and is gitignored;
  the committed template is `config.example.yaml`.
- Don't describe the output as full CAM: 3-axis, single tool, no roughing/finishing
  split, no collision control — README's "Ограничения" section is the honest contract.
