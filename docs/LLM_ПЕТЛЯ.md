# ЛЛМ в пайплайне: полный технический разбор

Как устроено автономное исправление программы обработки через ЛЛМ
(`auto_fix.py`): транспорт, форматы данных, промпт, контракт ответа,
валидация действий, пороги, безопасность.

```
┌────────────┐   ┌────────────┐   ┌──────────────┐   ┌──────────────┐
│ генерация  │ → │ симуляция  │ → │  булев diff  │ → │  claude -p   │
│ (FreeCAD)  │   │ (NX ISV)   │   │ (недорез/    │   │  (ЛЛМ)       │
│            │   │            │   │  зарез→JSON) │   │              │
└────────────┘   └────────────┘   └──────────────┘   └──────┬───────┘
      ↑                                                     │ JSON-действия
      └─────────── set_param / extra_zone / skip_op / dead_zone ──┘
```

Одна итерация = генерация (~10–60 с) + симуляция (~2–3 мин) + diff (~30–60 с)
+ запрос ЛЛМ (~30–90 с). Петля крутится до `--iters` (дефолт 3), «дефектов нет»
или вердикта «параметрами не лечится».

---

## 1. Транспорт: как зовём ЛЛМ

**Без API-ключей.** Используется headless-режим Claude Code CLI:

```
claude -p [--model <model>]     # промпт подаётся через stdin
```

- Работает по подписке Claude (та же учётка, что у Claude Code / VSCode-расширения).
- Поиск бинарника (`auto_fix.find_claude()`), по приоритету:
  1. переменная окружения `CLAUDE_CLI` (явный путь);
  2. `claude` / `claude.cmd` / `claude.exe` в `PATH` (терминальный Claude Code);
  3. бинарник VSCode-расширения:
     `~/.vscode/extensions/anthropic.claude-code-*/resources/native-binary/claude.exe`
     (берётся свежайшая версия расширения).
- Вызов: `subprocess.run([exe, "-p", "--model", m], input=промпт, timeout=900)`,
  кодировка UTF-8, `errors="replace"`.
- Выбор модели: флаг `--llm-model` у `auto_fix.py`. Дефолт `opus`; варианты:
  `sonnet`, `claude-fable-5`, пустая строка = дефолт сессии CLI. Эскалация
  (например, opus → fable) — ручная, перезапуском с другим флагом.

**Почему CLI, а не API:** нет ключей и биллинга по токенам, единая подписка,
модель и аутентификация управляются самим Claude Code.

## 2. Входные данные для ЛЛМ (что кладём в промпт)

Промпт собирает `auto_fix.build_prompt()` из четырёх машинных источников.

### 2.1 Описание детали — `step_describe.py` → JSON

Сырой STEP для ЛЛМ непригоден (граф сущностей, тысячи координат). Воркер
`freecad_describe_worker.py` (внутри freecadcmd) сводит его к компактному JSON:

```json
{
 "bbox": {"x": [0, 67], "y": [0, 40], "z": [-24, 0], "size_mm": [67, 40, 24]},
 "volume_cm3": 7.2,
 "faces": [{"type": "plane", "area_mm2": 2047, "normal": [0, 0, 1],
            "bbox": {...}}, ...],
 "holes": [{"type": "through", "diameter_mm": 14.0, "depth_mm": 2.5,
            "center_xy": [46.85, 25.0], "z_range": [-21.5, -19.0]}],
 "summary": "Тело 67×40×24 мм, объём 7.2 см³, 21 грань; ..."
}
```

Важно: описывается `<out>_part.step` — деталь **в координатах программы**
(после auto-orient и сдвига нуля), той же СК, что G-код и симуляция. Описание
считается один раз и кэшируется на все итерации.

### 2.2 Расхождение — `step_diff.py` → JSON

Булево сравнение «деталь vs фактический вырез» (`freecad_diff_worker.py`):

```json
{
 "part_volume_mm3": 6867.5,
 "result_volume_in_part_footprint_mm3": 7442.1,
 "floor_clearance_mm": 0.5,
 "floor_skin_mm3": 574.6,            // намеренная плёнка от стола — НЕ дефект
 "tessellation_noise_mm3": 36.1,     // шум фасетирования — отброшен
 "undercut_total_mm3": 0,            // недорез: осталось, где не должно
 "overcut_total_mm3": 0,             // зарез: снято, что должно остаться
 "undercuts": [{"volume_mm3": ..., "center": [x, y, z],
                "size": [dx, dy, dz], "bbox": {...}}, ...],  // топ-15 по объёму
 "overcuts": [...],
 "note": "пояснение семантики полей для ЛЛМ"
}
```

Технические грабли, учтённые внутри (ЛЛМ их не видит, но без них данные — мусор):
- фасетное тело из NX «перестирывается» через меш
  (tessellate → makeShapeFromMesh → makeSolid, reverse при инверсии);
- булевы OCCT с фасетным телом надёжны только когда оно **правый** операнд:
  недорез = `(bbox − деталь) − ((bbox − деталь) − результат)`;
- намеренная плёнка дна отрезается слоем `z < дно + FLOOR_CLEARANCE`;
- «диффузные плёнки» тесселяции (заполнение < 2 % собственного bbox) — шум;
- рамка заготовки вне силуэта детали не учитывается (остаётся по техпроцессу).

### 2.3 Состояние генератора

- текущие значения всех параметров из белого списка (см. §4);
- список операций программы, распарсенный из комментариев G-кода
  `(Begin|Finish operation: X)` — чтобы ЛЛМ могла ссылаться на операции по имени;
- активные `DEAD_ZONES` / `EXTRA_ZONES` / `SKIP_OPS`.

### 2.4 История итераций

```json
[{"iter": 1, "undercut_mm3": 56148.3, "overcut_mm3": 21.8,
  "actions": [{"extra_zone": {...}, "reason": "..."}]}]
```

История — ключ к сходимости: по ней модель видит «действие не изменило diff
до сотой» и пересматривает гипотезу, а не повторяет её (проверено живьём:
итерация 2 на детали 033_B).

## 3. Промпт (структура)

Один user-промпт (система не используется), по-русски. Секции по порядку:

1. **Роль и контекст**: «технолог-программист ЧПУ», 3-осевая фрезеровка сверху,
   одна концевая фреза Ø текущий.
2. **ОПИСАНИЕ ДЕТАЛИ** — JSON из §2.1 (одной строкой, без pretty-print).
3. **РАСХОЖДЕНИЕ** — JSON из §2.2.
4. **ТЕКУЩИЕ ПАРАМЕТРЫ** + операции + зоны из §2.3.
5. **ИСТОРИЯ** из §2.4 с указанием «не повторяй испробованное без причины».
6. **ВАЖНЫЕ ФАКТЫ** — прививка от ложных срабатываний:
   - floor_skin намеренный (зазор от стола), чинить не надо;
   - рамка заготовки — по техпроцессу;
   - припуск управляется `ROUGH_ALLOWANCE_MODE`;
   - недоступное сверху (поднутрения) 3-осевой не лечится → `unfixable`;
   - фрезу МОЖНО менять (`TOOL_DIAMETER`, «на складе есть любая»), с физикой:
     меньше Ø → лезет в узкие места, но режет медленнее.
7. **ЗАДАЧА + список действий** с их сигнатурами (см. §4).
8. **Требование к ответу**: «СТРОГО ОДИН JSON-объект без markdown», схема ниже.

## 4. Контракт ответа и действия

### Схема ответа

```json
{"analysis": "краткий разбор по-русски",
 "verdict": "ok | retry | unfixable",
 "actions": [...],
 "report": "итог для оператора"}
```

Парсинг (`extract_json`): из текста берётся первый **сбалансированный** блок
`{...}` (скобочный счётчик) — модель может обернуть JSON текстом или ```-блоком,
это не ломает разбор. Невалидный JSON → лог + стоп петли (сырой ответ пишется
в журнал).

### Действия (все проходят валидацию на стороне Python)

| Действие | Сигнатура | Лечит | Валидация |
|---|---|---|---|
| `set_param` | `{"type":"set_param","name":N,"value":V}` | системные проблемы | имя строго из белого списка; float клампится в границы; str — из перечня |
| `extra_zone` | `{"type":"extra_zone","x":[x0,x1],"y":[y0,y1],"z_bottom":z,"reason":R}` | недорез по месту | float-каст; зона режется Adaptive (fallback Profile-Inside), клампится полом и мёртвыми зонами |
| `skip_op` | `{"type":"skip_op","name":"RoughSlope2","reason":R}` | зарез от конкретной операции | имя по regex `(RoughHole\|RoughFace\|RoughSlope\|ExtraZone)\d+\|RoughPerimeter\|Finish` |
| `dead_zone` | `{"type":"dead_zone","x":[..],"y":[..],"reason":R}` | зарез по месту (крайняя мера) | float-каст; вычитается из всех 2D-зон, наклонные с центром в зоне пропускаются |

### Белый список `set_param` (границы клампятся молча)

| Параметр | Тип | Границы |
|---|---|---|
| `TOOL_DIAMETER` | float | 1.0–20.0 мм |
| `ROUGH_STEPDOWN` | float | 0.2–3.0 мм |
| `ROUGH_STEPOVER` | float | 10–60 % |
| `ROUGH_TOLERANCE` | float | 0.05–0.3 мм |
| `ROUGH_ALLOWANCE` | float | 0–1 мм |
| `ROUGH_ALLOWANCE_MODE` | str | none / xy / all |
| `ROUGH_MODE` | str | stages / layers |
| `FINISH` | bool | — |
| `SURFACE_STEPOVER` | float | 10–50 % |
| `SURFACE_SAMPLE_INTERVAL` | float | 0.2–1.0 мм |

Применение — прямой `setattr(config, ...)`: следующая генерация читает конфиг
заново, симуляция подхватывает новую фрезу автоматически (`nx_sim.write_to_ini`
перезаписывает таблицу инструментов стойки перед каждым прогоном).

## 5. Управление петлёй

- **Пороги «дефектов нет»** (до вызова ЛЛМ, мм³): `OK_UNDERCUT_MM3 = 100`,
  `OK_OVERCUT_MM3 = 10` — при выполнении обоих петля завершается без ЛЛМ.
- **Вердикты ЛЛМ**: `ok` → стоп (приемлемо); `retry` → применить действия,
  новая итерация; `unfixable` → стоп с объяснением (второй установ и т.п.).
- Защита от зацикливания: `retry` без применимых действий → стоп;
  лимит `--iters`.
- **Журнал** `<gcode>_autofix.json`: по итерации — строки G-кода, машинное
  время, объёмы недореза/зареза, wall-время, `analysis/verdict/report` ЛЛМ,
  применённые действия; при не-JSON ответе — сырой текст (первые 2000 симв.).

## 6. Уровни безопасности

1. **ЛЛМ не пишет координаты траектории** — только параметры и зоны;
   координаты считает FreeCAD.
2. **Белый список с границами** — вне списка/границ отклоняется или клампится.
3. **Зазор от стола** (`FLOOR_CLEARANCE`, дефолт 0.5 мм) зашит в
   `make_adaptive/make_profile/make_surface_rough`: НИ одна операция, включая
   `extra_zone` от ЛЛМ, не опустится ниже «дно детали + зазор».
4. **skip_op** валидируется по шаблону имён — отключить можно только операцию,
   произвольную строку в конфиг не протащить.
5. Смена инструмента реальна только в симуляции; на станке фрезу ставит
   оператор — рекомендация фиксируется в `report`/журнале.

## 7. Проверенный кейс (деталь 75.6121.0.0411.033_B, opus, 3 итерации)

| Итерация | Diff | Решение ЛЛМ |
|---|---|---|
| 1 | недорез **56 148 мм³** (карман не выбран штатными операциями) | `extra_zone` над карманом |
| 2 | без изменений (до 0.01) | по истории понял: путь пуст, Ø12 не влезает в спираль врезания → `TOOL_DIAMETER = 6` + повтор `extra_zone` |
| 3 | **недорез 0, зарез 0**, плёнка дна 182 мм³ | стоп ✅ |

Цепочка «диагноз → гипотеза → наблюдение → пересмотр причины → смена
инструмента → сходимость» — целевое поведение петли.

## 8. Ограничения и план

- Одна фреза на программу (ограничение генератора, не петли).
- Пороги (100/10 мм³) — эвристика, не привязана к допускам чертежа.
- ЛЛМ не видит картинку — только числа; рендер результата в промпт (vision) —
  следующий кандидат.
- Машинное время не оптимизируется (петля чинит геометрию, не скорость).
- Индексы граней между `step_describe` и воркером не связаны — действия
  «по конкретной грани» пока невозможны, только «по зоне».
