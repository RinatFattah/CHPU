# CAM: CAD-модель → G-Code (FreeCAD)

Получение траектории обработки **по готовой модели детали**, без LLM и Blender.
Вход — CAD-файл, выход — управляющая программа (G-Code) для станка с ЧПУ.
Стратегия — **3D-обработка по поверхности**: фреза следует за фактической геометрией
модели (наклоны, конусы, купола, рельеф).

```
деталь.prt (Siemens NX)
   │  экспорт из NX: File → Export → STEP (AP214/AP242)
   ▼
деталь.step ──► python run_cam.py деталь.step ──► деталь.gcode
                        (FreeCAD Path/CAM, headless)
```

## Форматы входа

| Формат | Как читается | Рекомендация |
|---|---|---|
| `.step` / `.stp` | точное твёрдое тело (BREP), единицы из файла | ✅ **основной вход** |
| `.iges` / `.igs`, `.brep` | точное тело | ✅ да |
| `.stl` / `.obj` | меш → фасеточное тело (аппроксимация) | допустимо, хуже точность |
| `.prt` (Siemens NX) | **не читается** — закрытый формат | ❌ экспортируйте STEP из NX |

## Быстрый старт

```bash
python run_cam.py detal.step                          # черновая + чистовая (дефолт)
python run_cam.py detal.step out.gcode --config config.yaml
python run_cam.py detal.step --no-finish              # только черновая
python run_cam.py detal.step --stock-margin 0         # заготовка точно по детали
python run_cam.py detal.step --stock otlivka.step     # заготовка из файла (отливка)
python run_cam.py detal.step --verify-export          # + эталон и маски для verify.py
python run_cam.py scan.stl --mm                       # меш в миллиметрах
```

Флаги CLI: `--config FILE`, `--rough MM` (припуск черновой), `--no-finish`,
`--stock-margin MM`, `--stock FILE`, `--origin ...`, `--no-orient`,
`--nx-export` (STEP деталь/заготовка для NX), `--verify-export` (эталон+маски для
[`verify.py`](#верификация-детали-после-симуляции-verifypy)); для мешей — `--mm` / `--meters` /
`--scale N` (STEP/IGES масштаб не нужен — единицы в файле).

## Как строится обработка

Черновая идёт **по этапам, как у технолога** — отдельными операциями в порядке
техпроцесса, зоны не пересекаются (фреза не проходит одно место дважды):

1. **RoughContour** — контур: материал между силуэтом детали и краем заготовки.
   Существует и при `STOCK_MARGIN` = 0: заготовка — всегда бокс, и если деталь
   в плане не прямоугольник (скругления углов, выступы), по периметру есть
   материал. Широкие поля (≥ 2 диаметров фрезы) выбираются адаптивно; узкие —
   контурным проходом: фреза обходит силуэт с отступом радиус + припуск,
   слой за слоем (заход не нужен — идёт по воздуху вокруг заготовки).
   Промежуточные поля (шире прохода фрезы, но уже 2 диаметров) оставят
   гребень по периметру — об этом печатается предупреждение.
2. **RoughHole1..N** — **сквозные вырезы любой формы**, по очереди от ближнего
   к дальнему. Вырез = внутренний замкнутый контур силуэта (круглый, овальный,
   фигурный — не важно), режется одной зоной. Куда фреза с припуском не влезает —
   пропуск с сообщением в лог.
3. **RoughFace1..N** — **каждая плоская грань отдельной операцией** («контуры»,
   которые видно в NX): полки, донья карманов, уступы. Порядок: сверху вниз,
   на одном уровне — от ближней. Глубина каждой операции = высота грани + припуск.
4. **RoughSlope1..N** — **каждая наклонная/криволинейная грань** (скос, купол,
   радиус гиба) **отдельной операцией, «террасами»**: фреза следует за
   поверхностью слоями `ROUGH_STEPDOWN`, держась на `ROUGH_ALLOWANCE` выше неё.
   Берутся грани, у которых есть площадь в проекции на основание (нормаль хоть
   немного смотрит вверх) и над которыми нет материала. Плоская фреза оставляет
   на наклоне ступеньки — их снимает чистовой проход.

Чего черновая сознательно НЕ делает: **недоступное сверху** (грани, смотрящие
вниз/вбок, материал под нависающими элементами) — это второй установ, программа
честно его пропускает (площадь печатается предупреждением), а не режет сквозь
деталь.

Этапы 1–3 — адаптивная выборка (FreeCAD Path Adaptive) слоями `ROUGH_STEPDOWN`
с припуском `ROUGH_ALLOWANCE` по стенкам и по дну; этап 4 — проход по поверхности
(Path Surface, Multi-pass). После черновой по умолчанию идёт **чистовой проход**
по поверхности — отключается `--no-finish` / `FINISH: false`.

**Экспериментальный режим `--rough-mode layers`** (`ROUGH_MODE: "layers"`) —
послойная черновая, как Cavity Mill в NX: материал на высоте Z = сечение
заготовки − тень детали выше Z. Высоты делятся на диапазоны между характерными
уровнями детали (наклонные участки дробятся по `ROUGH_STEPDOWN`), соседние
диапазоны с одинаковой зоной склеиваются, каждая связная область — отдельная
операция (`B<N>Contour`/`B<N>Pocket`, сверху вниз, на уровне — от ближней).
Ничего не предполагает о форме детали и заготовки, поэтому устойчивее к
нестандартным заготовкам (уголок, отливка) и обычно быстрее по машинному
времени; этапный режим остаётся дефолтом.

**Заготовка** — два варианта:
- **бокс** (по умолчанию): габарит детали + поля (`STOCK_MARGIN` по X/Y,
  `STOCK_MARGIN_TOP` сверху);
- **произвольная из файла** (`STOCK_FILE` / `--stock FILE`, форматы те же, что
  у модели): отливка, поковка, обрезок. Файл должен быть **в той же системе
  координат, что и модель детали** (например, оба экспортированы из одной
  сборки NX) — заготовка автоматически поворачивается и сдвигается синхронно
  с деталью. Зона контура при этом строится от реального силуэта заготовки.
  Если деталь выступает из заготовки — предупреждение в лог.
  Заготовка не обязана иметь ровный верх (уголок, отливка): каждая операция
  сама находит локальный верх материала над своей зоной и начинает резать
  оттуда, а зоны, над которыми материала нет, пропускаются. Материал над
  самым верхом детали тоже снимается; узкие верхние полки (уже фрезы)
  обрабатываются террасами по поверхности.

Размеры заготовки, детали и инструмента печатаются в лог и в шапку G-Code
комментариями (`(Stock: ...)`, `(Part: ...)`, `(Tool: ...)`; координаты бокса —
в нуле программы, удобно копировать в настройку заготовки CAMotics).
Инструмент — один на программу: плоская концевая фреза `TOOL_DIAMETER`.

## Параметры генерации G-Code

Задаются в `config.yaml` (копия `config.example.yaml`) или в `config.py`.
Приоритет: дефолты → YAML из `--config` → флаги CLI.

### 1. Инструмент и режимы резания — задать ОБЯЗАТЕЛЬНО под свой станок/материал

| Параметр | Дефолт | Ед. | Что означает |
|---|---|---|---|
| `TOOL_DIAMETER` | `6.0` | мм | диаметр фрезы. Для криволинейных поверхностей желательна **сферическая (ball-nose)** — плоская оставляет ступеньки между строчками |
| `FEED_RATE` | `800` | мм/мин | рабочая подача; вертикальная врезка автоматически = 1/4 |
| `SPINDLE_SPEED` | `12000` | об/мин | обороты шпинделя (M3 S…) |
| `SAFE_HEIGHT` | `10.0` | мм | высота над заготовкой для холостых перемещений |

### 2. Заготовка

| Параметр | Дефолт | Ед. | Что означает |
|---|---|---|---|
| `STOCK_MARGIN` | `12.0` | мм | поля заготовки вокруг детали по X и Y. Дефолт = 2 диаметра фрезы (при таких полях контур выбирается адаптивно). `0` = заготовка по габариту детали, контур — одним проходом вокруг силуэта (углы, выступы) |
| `STOCK_MARGIN_TOP` | `0.0` | мм | запас материала над деталью (Z+) |
| `STOCK_FILE` | `""` | — | заготовка из файла (`--stock`); пусто = бокс. Файл в той же системе координат, что и модель; `STOCK_MARGIN` игнорируется |

### 3. Черновая обработка

| Параметр | Дефолт | Ед. | Что означает |
|---|---|---|---|
| `ROUGH_MODE` | `"stages"` | — | `stages` = по типам фич; `layers` = послойно, как Cavity Mill (эксперимент, `--rough-mode layers`) |
| `ROUGH_ALLOWANCE` | `0.5` | мм | **припуск**: сколько материала оставить под чистовую, шаг 0.1. `0` = черновая выключена |
| `ROUGH_STEPDOWN` | `1.0` | мм | слой съёма за проход (главный рычаг времени обработки) |
| `ROUGH_STEPOVER` | `40` | % Ø | шаг адаптивной строчки |
| `ROUGH_TOLERANCE` | `0.1` | мм | точность расчёта траектории; для черновой 0.2 — быстрее почти без потерь |

### 4. Чистовая

| Параметр | Дефолт | Ед. | Что означает |
|---|---|---|---|
| `FINISH` | `true` | — | чистовой 3D-проход по поверхности после черновой (отключить: `--no-finish`) |
| `SURFACE_STEPOVER` | `40` | % Ø | шаг между строчками. Чистовая: 10–25 % |
| `SURFACE_SAMPLE_INTERVAL` | `0.5` | мм | разрешение сэмплирования поверхности |
| `SURFACE_CUT_PATTERN` | `"Offset"` | — | рисунок строчек: `Offset`, `ZigZag`, `Line`, `Spiral`, `Circular` |

### 5. Вывод и система

| Параметр | Дефолт | Что означает |
|---|---|---|
| `ORIGIN` | `"corner-top"` | ноль программы: `corner-top` — X0 Y0 в углу габарита детали, Z0 = верх (стандарт ЧПУ); `center-top` — X0 Y0 в центре; `model` — не сдвигать (как в CAD-файле). Детали из NX часто экспортированы в координатах сборки — без нормализации фреза уедет «в бок» на первое же перемещение |
| `POSTPROCESSOR` | `"grbl"` | диалект G-Code под контроллер: `grbl`, `linuxcnc`, `fanuc`, `mach3_mach4`, `marlin`, `smoothie`, `heidenhain`, `centroid`, `uccnc` и др. |
| `FREECAD_CMD` | `""` | путь к `freecadcmd`; пусто = автопоиск (AppImage → PATH → snap) |
| `FREECAD_TIMEOUT` | `3600` | сек — лимит на расчёт траектории (черновая Adaptive на сложной детали считается десятки минут) |
| `STL_SCALE_TO_MM` | `1000.0` | **только для мешей**: множитель единиц → мм (Blender-метры = 1000, CAD-мм = 1.0). Для STEP/IGES игнорируется |

### Пример `config.yaml` под задачу

```yaml
TOOL_DIAMETER: 4.0          # фреза, которая реально стоит в шпинделе
FEED_RATE: 600              # подача под материал
SPINDLE_SPEED: 18000
POSTPROCESSOR: "linuxcnc"   # диалект вашей стойки
SURFACE_STEPOVER: 20        # чистовая обработка
SURFACE_SAMPLE_INTERVAL: 0.3
```

## Что проверить перед станком

1. **Габариты** — размеры траектории соответствуют модели.
2. **Симуляция** — прогнать в [CAMotics](https://camotics.org) или
   [ncviewer.com](https://ncviewer.com): траектория, глубины, отсутствие зарезов.
3. **Диалект** — `POSTPROCESSOR` соответствует контроллеру станка.
4. **Ноль детали** — по умолчанию (`ORIGIN: corner-top`) X0 Y0 = левый-ближний угол
   габарита детали, Z0 = её верхняя плоскость; выставить ноль на станке соответственно.
5. Контроля столкновений и зарезов **нет** — ответственность на операторе.

## Верификация детали после симуляции (`verify.py`)

Отдельный инструмент: сравнивает деталь **после симуляции обработки** с эталоном и
проверяет, укладывается ли отклонение в припуск. Симуляция — внешняя (NX ISV или
CAMotics); из неё выгружается результат как STL/STEP.

```bash
# 1) пайплайн: G-код + эталон и маски достижимости (всё в СК G-кода)
python run_cam.py detal.step out.gcode --stock zag.step --verify-export
# 2) прогнать out.gcode в симуляторе, выгрузить результат как machined.stl
# 3) верификация по припуску
python verify.py machined.stl --from-export out --allowance 0.5
```

`--verify-export` кладёт рядом с G-кодом `out_part.stl` (эталон, watertight),
`out_reachable.stl` и `out_unreachable.stl` (грани, недостижимые с этого установа).
`--from-export out` подхватывает все три автоматически.

Что считается и критерий (припуск = верхняя граница):
- **зарез** — материал снят *ниже* номинала — брак, должен быть ≤ `--gouge-tol` (0.05 мм);
- **избыток** — материал стоит *выше* номинала — это и есть припуск, должен быть
  ≤ `--allowance`, и проверяется **только на достижимых поверхностях** (зоны «второго
  установа» исключаются по маске, но показываются на карте серым).

**PASS ⇔ нет зареза И избыток ≤ припуска.** Выход: сводка в консоль, цветная карта
отклонений `machined_deviation.ply` (зелёный — в припуске, красный — избыток, синий —
зарез, серый — второй установ) и, по `--json`, отчёт; код возврата **0** (PASS) /
**2** (FAIL) — удобно в скриптах.

| Флаг/параметр | Дефолт | Что означает |
|---|---|---|
| `machined` | — | результат симуляции (STL/STEP), позиционный аргумент |
| `--allowance MM` | — | **припуск**: верхняя граница избытка (обязательный) |
| `--gouge-tol MM` | `0.05` | допустимый зарез ниже номинала |
| `--from-export BASE` | — | взять эталон+маски из `BASE_part.stl` / `_reachable.stl` / `_unreachable.stl` |
| `--nominal FILE` | — | эталон (STL/STEP), если не через `--from-export` |
| `--reachable` / `--unreachable FILE` | — | маски достижимости; без них проверяется вся поверхность |
| `--out FILE` | `<machined>_deviation.ply` | карта отклонений |
| `--json FILE` | — | машиночитаемый отчёт |

Оба входа обязаны быть в **одной системе координат** — для этого и эталон
(`--verify-export`), и симуляция берут СК G-кода. Ограничение точности маскирования:
у самой границы достижимых/недостижимых зон отдельные точки могут отнестись не в ту
сторону — на плотном результате симуляции это тонкая полоса, на вердикт влияет слабо.
Зависимости: `numpy`, `scipy`, `trimesh` (см. `requirements.txt`); STL читается
напрямую, STEP — тесселлируется через FreeCAD.

## Ограничения

- 3-осевая обработка сверху: поднутрения, обратная сторона, боковые элементы недоступны.
- Обрабатывается то, что «видно сверху»; вертикальные стенки получаются проходами строчек.
- Один инструмент на программу; черновая и чистовая не разделяются.
- Крепёж/оснастка не моделируются; заготовка = габарит модели.
- `.prt` требует ручного экспорта STEP из NX (открытых конвертеров не существует).

## Как это устроено внутри

- `run_cam.py` — CLI-обёртка.
- `freecad_cam.py` — хост: находит `freecadcmd`, передаёт параметры (JSON через
  env `FREECAD_WORKER_PARAMS`), парсит результат. FreeCAD — отдельный процесс.
- `freecad_worker.py` — исполняется внутри FreeCAD: модель → тело → Path Job
  (заготовка + фреза) → операция Surface → постпроцессор → G-Code. Под
  `--verify-export` дополнительно пишет эталон и маски достижимости граней в STL.
- `verify.py` — самостоятельный (numpy/scipy/trimesh, FreeCAD не нужен): результат
  симуляции ↔ эталон, проверка по припуску.

Установка FreeCAD (AppImage, без root) и быстрый старт — в основном [README.md](README.md).
